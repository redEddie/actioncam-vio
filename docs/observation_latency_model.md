# 관측 지연·지터 노이즈 모델 (VLA 학습용)

VLA 롤아웃에서 정책이 보는 이미지는 "지금"이 아니라 **지연 L만큼 과거**의
세계이고, L은 프레임마다 흔들린다(지터). action chunking이 지연의 영향을
완화하지만, **학습 시 같은 통계의 지연/지터를 주입**하면(sim2real 노이즈)
train-rollout 격차를 더 줄일 수 있다. 이 문서는 실측값과 그것을 재현하는
노이즈 모델을 기록한다 (측정: `gopro_vio/latency_test.py`).

## 1. 실측값 (glass-to-host, 모니터 응답 포함)

**현행 롤아웃 경로 (2026-07-13, 측정도구 v3):**

| 경로 | 조건 | median | p10 | p90 | 지터 폭(p10–p90) | 샘플 |
|---|---|---|---|---|---|---|
| **HERO7 + AVerMedia GC553Pro** | 1080p60 **YUYV(무압축)** | **111 ms** | 94 | 131 | **37 ms** | 615 |

구형 경로 (2026-07-07, 측정도구 v1 — 아래 주의 참조):

| 경로 | 조건 | median | p10 | p90 | 지터 폭 |
|---|---|---|---|---|---|
| Ace Pro 2 UVC (반납됨) | 1080p30 MJPG | 163 ms | 145 | 185 | 40 ms |
| HERO7 + MacroSilicon(MBF) 동글 (교체됨) | 720p60 MJPG | 186 ms | 131 | 227 | 96 ms |

⚠️ **측정도구 버전 주의**: v1은 캡처·표시·무거운 처리(검출/인코딩)가 한
루프에 있어 (a) 드라이버 버퍼에서 묵은 프레임을 읽고 (b) 화면 패턴 자체가
묵은 채 찍혀, **수십 ms 과대 추정**이었다. 같은 GC553Pro를 도구 버전별로
재면 263(v1 상당) → 189(캡처만 분리) → 111 ms(완전 분리)로 줄어드는 것을
실측했다. v3는 캡처 스레드(도착 즉시 타임스탬프) + 처리 워커 + 표시 전용
루프로 분리되어 잔여 편향이 vsync 양자화(~8 ms)뿐. Ace는 반납되어 v3
재측정이 불가능하므로 구수치를 각주와 함께 보존한다. 구경로 참고 특성:
MBF 동글은 1080p30 인접 프레임 블렌딩(이중상), 1080p60 프레임 손상이
있었다 — GC553Pro는 무압축 1080p60이 정상이라 해당 없음.

## 2. 측정치 → 카메라 파이프라인 지연 (보정)

측정값에는 실제 롤아웃에 없는 두 성분이 포함된다: 모니터 응답(~15 ms)과
패턴 나이(v3 기준 vsync 양자화 ~8 ms). 이를 빼면:

| 경로 | 롤아웃 지연 상수 μ | 지터 표준편차 σ* |
|---|---|---|
| **HERO7 + GC553Pro (현행)** | **≈88 ms** | **≈14 ms** |
| Ace Pro 2 UVC (v1 측정 — 과대 포함) | ≈132 ms | ≈13 ms |
| HERO7 + MBF 동글 (v1 측정 — 과대 포함) | ≈163 ms | ≈35 ms |

*p10–p90 폭을 정규분포로 환산(폭≈2.56σ).

## 3. 노이즈 모델

프레임별 관측 지연을 다음으로 샘플링한다:

```
L_t = μ + ε_t,   ε_t ~ N(0, σ²),   L_t를 [μ-3σ, μ+3σ]로 클립
```

| 경로 | 모델 |
|---|---|
| **HERO7+GC553Pro (현행)** | **L ~ N(88, 14²) ms** |
| Ace Pro 2 (참고, v1) | L ~ N(132, 13²) ms |
| HERO7+MBF (참고, v1) | L ~ N(163, 35²) ms |

30 fps 학습 데이터에서는 프레임 인덱스 시프트로 구현된다
(1프레임 = 33.4 ms):

```python
import numpy as np

def delayed_frame_index(t_idx, fps=30.0, mu_ms=88.0, sigma_ms=14.0,
                        rng=np.random):
    """학습 샘플의 관측으로 '몇 프레임 전' 이미지를 쓸지 샘플링.

    action label은 t_idx 시점 그대로 두고, 이미지 관측만 과거로 민다.
    """
    L = np.clip(rng.normal(mu_ms, sigma_ms), mu_ms - 3*sigma_ms,
                mu_ms + 3*sigma_ms)
    return max(0, t_idx - int(round(L * fps / 1000.0)))
```

- HERO7+GC553Pro(88±14 ms): 30 fps 기준 시프트 3프레임 ± 0~1프레임 —
  지터가 프레임 양자화 수준이라 **상수 3프레임** 근사도 무방.
  60 fps 데이터라면 5프레임 ± 1프레임.
- 에피소드 단위 상수 성분의 느린 변동(온도/부하)은 미측정 — 에피소드마다
  μ 자체를 ±10 ms 재샘플링하는 계층적 노이즈로 보수적으로 커버 가능.

## 4. 함께 주입할 만한 경로별 아티팩트 (선택)

| 아티팩트 | 대상 경로 | 증강 구현 |
|---|---|---|
| 기하 정합 | HERO7+GC553Pro는 **크롭만** (필러박스 4:3, `frame[:, 240:1680]` → 녹화 기하 ×0.5325) — 어안 유지, 워프 불필요. 타 경로는 `gopro_vio.remap` |
| 압축/색 변환 | GC553Pro는 무압축 YUYV — MJPEG류 증강 불필요. 구 MJPG 경로만 JPEG 재인코딩 60~90 |
| 프레임 블렌딩 | 구 MBF 1080p30 한정 (`I' = 0.5·I_t + 0.5·I_{t-1}`) — 현행 경로 해당 없음 |

적용 순서: **remap(기하) → 지연 시프트(시간) → 화질 노이즈**.

## 5. 하드웨어가 바뀌면

캡처보드나 카메라 교체 시 μ, σ만 재측정해 갱신 (무압축 지원 보드는
`--fourcc YUYV` 권장, 통계 확보를 위해 90초 이상):

```bash
python -m gopro_vio.latency_test <장치 by-id 경로> --size 1920x1080 --fps 60 \
    --fourcc YUYV --seconds 120 --setup
```

측정 시 주의: **모니터는 한 대만** (복제 모드면 마커 id가 중복 검출되어
디코딩 실패), 뷰어 버퍼링 지연과 혼동 금지. 측정 원리는
`latency_test.py` docstring 참조.
